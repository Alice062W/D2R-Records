# Category-Grid Icons — Design

## Goal

Show a representative item icon on each category tile in `CategoryCardGrid` (currently
plain text), matching d2r.world's category-grid presentation (e.g.
`https://d2r.world/en-US/info/item/unique` shows a generic item-type icon per tile).

## Background

`CategoryCardGrid` is reused by Base Items, Unique Items, Set Items, Magic Items, and
Rare Items landing pages. Its `categories` prop is a list of category slugs — for
Base/Unique/Set these are always the 28 canonical `SLOT_ORDER` slugs
(`src/lib/grail/catalog.ts:107`); for Magic/Rare Items these are the 37 real slugs found
in `data/magic-affixes.json` (25 of which are the same canonical slugs, plus 12
generic/class-fallback codes — `armo`, `bar`, `blun`, `h2h`, `mele`, `miss`, `rod`,
`shld`, `staff`, `thro`, `weap`, `amaz` — that don't correspond to a real placeable item
and are a separate, already-flagged taxonomy problem, out of scope here).

Investigated icon availability for all 28 canonical categories: 24 already have a
representative entry in `data/bases-full.json` whose `invFile` (looked up via
`vendor/d2data/json/items.json`) exists in the 623 self-extracted icons. The remaining 4
(`rings`, `amulets`, `charms`, `jewels`) have no grade-tiered base-item entry in
`bases-full.json` at all (they aren't graded normal/exceptional/elite), plus `katars`
needs a different lookup (its base entries use `h2h` as the D2 item *type* code, not a
`bases-full.json` slotCategory match) — all 5 resolve via a small explicit code map
instead (`rin`→Ring, `amu`→Amulet, `cm1`→Small Charm, `jew`→Jewel, `ktr`→Katar), each
verified to have an existing icon.

## Non-goals

- The 12 generic/class-fallback Magic/Rare categories get NO icon in this pass — they
  keep today's text-only tile. Giving them an icon would require first fixing the
  underlying category-taxonomy gap (confirmed against d2r.world's real Magic Items page:
  it has granular categories like "Barbarian Helms"/"Circlets"/"Amazon Spears" that our
  generator currently collapses into these 12 generic buckets) — a separate,
  already-agreed follow-up project. This design's icon lookup simply has no entry for
  these 12 slugs, so `CategoryCardGrid` degrades to its current text-only rendering for
  them automatically (no special-casing needed).
- No change to the underlying category taxonomy/slug derivation itself.
- No icons on Runes/Cube Recipes/Crafted Items pages (no category-grid landing page
  exists for these — confirmed in the original Runes/Cube/Crafted design, still true).

## Design

**Data layer:** add a new generated file, `data/category-icons.json`, mapping each of the
28 canonical `SLOT_ORDER` slugs to an `invFile` string. Generated by
`scripts/generate-grail-data.mjs`:
- For 24 categories: look up the first `data/bases-full.json` entry with a matching
  `slotCategory`, take its `id` (strip the `base-` prefix to get the item code), and look
  up that code's `invfile` in `vendor/d2data/json/items.json`.
- For the 5 exceptions (`rings`, `amulets`, `charms`, `jewels`, `katars`): a small
  hardcoded `{ slotCategory: itemCode }` map (`rin`, `amu`, `cm1`, `jew`, `ktr`
  respectively), each looked up the same way via `items.json`.
- Every resolved `invFile` is verified at generation time to have a corresponding PNG in
  `public/items/inv/` (throw if not, so a future vendor-data change can't silently produce
  a broken icon reference).

**Component layer:** `CategoryCardGrid.tsx` imports `data/category-icons.json` and, for
each category tile, renders the mapped icon (same `object-contain`, `onError` fallback
pattern already established in `ItemStatCard`/`GrailItemDetail`) above or beside the label
if a mapping exists; renders exactly like today (label only) if it doesn't (covers the 12
generic Magic/Rare slugs automatically — no branching needed beyond "does this map have
this key").

## Testing plan

- Generator test: `data/category-icons.json` has exactly 28 entries (one per
  `SLOT_ORDER` slug), and every `invFile` value has a matching file in
  `public/items/inv/`.
- Component test: `CategoryCardGrid` renders an icon for a category present in the map
  (e.g. `helms`) and renders text-only (no `<img>`) for a category absent from it (e.g.
  `bar`), matching the existing icon components' `onError`/conditional-render
  conventions.
- Manual + browser verification: Base/Unique/Set Items landing pages show icons on every
  tile; Magic/Rare Items landing pages show icons on the 25 matching tiles and no icon
  (today's plain tile) on the 12 generic ones.
